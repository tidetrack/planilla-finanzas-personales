## Objetivo de la Sesión

Simplificar la arquitectura del sistema de monedas eliminando la gestión dinámica y hardcodeando un conjunto fijo de 5 monedas, removiendo toda la UI de configuración y reduciendo la complejidad del código.

## Contexto

### Problema Identificado

- Sistema con gestión dinámica de monedas (tabla `MONEDAS`, `MonedaService`, UI de configuración)
- Complejidad innecesaria para un conjunto pequeño y fijo de monedas
- UI de configuración que no se utilizaba frecuentemente
- Múltiples capas de abstracción para funcionalidad simple

### Decisión Clave

Hardcodear 5 monedas fijas: **ARS** (base), **USD**, **EUR**, **AUD**, **CNY**

## Bugs Encontrados y Resueltos

### Bug #1: "Tabla no configurada: CONFIG"

**Síntoma**: Error al ejecutar `updateExchangeRates()` 
**Causa**: Archivos `10_ConfigService.js` y `05_MonedaService.js` eliminados localmente pero todavía presentes en Apps Script editor 
**Solución**: Eliminación manual de archivos en Apps Script web editor 
**Líneas afectadas**: N/A (problema de sincronización)

### Bug #2: Property Mismatch `.moneda_id` vs `.id`

**Síntoma**: Rates no se guardaban, validación fallaba con `undefined` 
**Causa**: Código usaba `.moneda_id` en objetos `CURRENCIES` que tienen `.id` 
**Solución**: Cambio de todas las referencias (5 ubicaciones):

- `06_ExchangeRateService.js` líneas 261, 276, 277 (`fetchExchangeRatesFromAPI`)
- `06_ExchangeRateService.js` líneas 481, 492-493 (`saveDolarAPIRate`)

**Archivos modificados**:

```javascript
// ANTES
if (moneda.moneda_id === baseMoneda.moneda_id)
 base_moneda_id: arsMoneda.moneda_id;

// DESPUÉS
if (moneda.id === baseMoneda.id) base_moneda_id: arsMoneda.id;
```

### Bug #3: "Tabla no configurada: MONEDAS"

**Síntoma**: Error al guardar exchange rates desde ExchangeRate-API 
**Causa**: `checkMonedaExists()` en `04_DataValidation.js` validaba contra tabla `MONEDAS` eliminada 
**Solución**: Reescritura de `checkMonedaExists()` para validar contra `CURRENCIES` 
**Archivo modificado**: `04_DataValidation.js` líneas 37-45

## Lecciones Aprendidas

### 1. Sincronización Local vs Apps Script

**Problema**: Archivos eliminados localmente seguían en Apps Script 
**Aprendizaje**: En proyectos clasp/Apps Script, eliminar archivos explícitamente en ambos lados 
**Acción futura**: Verificar siempre el estado del editor web después de eliminaciones

### 2. Property Naming Consistency

**Problema**: Estructura de datos cambió pero referencias a propiedades no 
**Aprendizaje**: Buscar exhaustivamente todas las referencias con `grep` antes de refactorizar 
**Acción futura**: Usar TypeScript o JSDoc más estricto para detectar estos errores

### 3. Validaciones con Tablas Eliminadas

**Problema**: Funciones de validación seguían referenciando tablas eliminadas 
**Aprendizaje**: Las validaciones son puntos críticos que deben actualizarse junto con cambios de data model 
**Acción futura**: Crear checklist de "lugares a revisar" al eliminar entidades del sistema

### 4. Debugging Sistemático

**Técnica exitosa**: Agregar logs DEBUG temporales en cada paso de la función problemática 
**Resultado**: Permitió identificar exactamente dónde fallaba (dentro de `getConfig()`) 
**Acción futura**: Mantener esta técnica para troubleshooting complejo

## Decisiones Arquitectónicas (ADR Candidato)

### ADR-001: Hardcoding de Monedas

**Contexto**: Sistema usaba gestión dinámica de monedas con tabla `MONEDAS` y UI para agregar/editar

**Decisión**: Hardcodear 5 monedas fijas (ARS, USD, EUR, AUD, CNY) en constante `CURRENCIES`

**Razones**:

1. **Simplicidad**: El conjunto de monedas no cambia frecuentemente
2. **Reducción de complejidad**: Elimina capa completa de abstracción
3. **Mantenibilidad**: Un solo archivo (`00_Config.js`) contiene toda la configuración
4. **Performance**: No hay queries a BD para obtener monedas

**Consecuencias Positivas**:

- ~23% menos código en módulo de monedas
- Menos puntos de falla
- Más fácil de entender para nuevos desarrolladores
- Configuración centralizada

**Consecuencias Negativas**:

- Agregar nueva moneda requiere cambio de código (no UI)
- No hay historial de cambios de monedas en BD
- Menos flexible para multi-tenant scenarios

**Alternativas Consideradas**:

1. **Mantener sistema dinámico**: Descartado por complejidad innecesaria
2. **Config file externo (JSON)**: Descartado porque requiere parsing y no da beneficio real vs constante
3. **Enum TypeScript**: No aplicable (proyecto usa JavaScript sin compilar)

**Estado**: Implementado y verificado
